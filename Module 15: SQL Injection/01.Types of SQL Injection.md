# 🗂️ Types of SQL Injection
Attackers use various tricks and techniques to view, manipulate, insert, and delete data from an application’s database. Depending on the technique used, there are several types of SQL injection attacks. This section discusses the various types of SQL injection attacks. Attackers use SQL injection attacks in many different ways by corrupting SQL queries. <br> 
In an SQL injection attack, the attacker injects malicious code through an SQL query that can read sensitive data and even can modify (insert/update/delete) it. 

There are three main types of SQL injection: 
- **In-band SQL Injection:** An attacker uses the same communication channel to perform the attack and retrieve the results. In-band attacks are commonly used and easy-to-exploit SQL injection attacks. The most commonly used in-band SQL injection attacks are error-based SQL injection and UNION SQL injection.
- **Blind/Inferential SQL Injection:** Blind or inferential SQL injection is a type of attack where the database does not show error messages or direct query results. Instead, attackers send crafted SQL queries and observe the application’s behavior, often through true or false responses, to slowly infer the database structure and extract information.
- **Out-of-Band SQL Injection:** Attackers use different communication channels (such as database email functionality or file writing and loading functions) to perform the attack and obtain the results. This type of attack is difficult to perform because the attacker needs to communicate with the server and determine the features of the database server used by the web application.

## In-Band SQL Injection 
In in-band SQL injection, attackers use the same communication channel to perform the attack and retrieve the results. Depending on the technique used, there are various types of in-band SQL injection attacks. The most commonly used in-band SQL injection attacks are error-based SQL injection and UNION SQL injection. 

The different types of in-band SQL injection are as follows: 

### Error-based SQL Injection
  Error-based SQL injection happens when attackers send invalid inputs to trigger database errors. These error messages reveal details about the database, which the attacker uses to craft malicious SQL queries. By exploiting this, they can extract or compromise sensitive data from the application.

### System Stored Procedure 
A system stored procedure can become a serious security risk when user inputs are not properly sanitized before being used to build SQL queries. Stored procedures are meant to execute predefined SQL statements safely, but if they include dynamic SQL that directly concatenates user input, they can be exploited through SQL injection. This allows attackers to bypass authentication, steal sensitive information, or even take full control of the database.

For example, consider the stored procedure:
```sql
Create procedure Login @user_name varchar(20), @password varchar(20)  
As  
Declare @query varchar(250)  
Set @query = 'Select 1 from usertable Where username = ' + @user_name + ' and password = ' + @password  
exec(@query)  
Go
```
In this procedure, the query is constructed dynamically using values provided by the user. If an attacker enters the input:
```
Username: anyusername or 1=1'  
Password: anypassword  
```
the resulting SQL query becomes:
```sql
Select 1 from usertable Where username = anyusername or 1=1' and password = anypassword
```
Here, the condition `or 1=1` is always true, meaning the authentication check is bypassed. The attacker can log in without knowing the correct password, gaining unauthorized access.

This type of vulnerability demonstrates why input validation and parameterized queries are critical in stored procedures. Instead of directly embedding user input into SQL strings, secure coding practices like using prepared statements or parameter binding ensure that inputs are treated as data rather than executable code. Without these protections, stored procedures become an entry point for SQL injection attacks that can compromise the entire system.

### Illegal/Logically Incorrect Query 
Illegal or logically incorrect query attacks are a form of SQL injection where attackers deliberately send faulty input to the database in order to trigger error messages. These errors often reveal important details about the database structure, such as table names, column names, or data types, which can later be used to craft more precise and harmful attacks. By analyzing the database’s error responses, attackers can gradually map how the backend system is built.

For example, if an attacker enters the input `Bob"` in the username field, the query becomes:
```sql
SELECT * FROM Users WHERE UserName = 'Bob"' AND Password = 'xxx'
```
Since the quotation marks are mismatched, the database returns an error like:
```sql
Incorrect Syntax near 'Bob'. Unclosed quotation mark after the character string '' AND Password='xxx''.
```

This error message confirms that the query is vulnerable and exposes details about how the database processes input. With such information, the attacker can move on to discovering column names, extracting data, or escalating the attack further. This demonstrates how even small syntax errors, when unhandled, can provide attackers with valuable insights into the database structure.

### UNION SQL Injection 
UNION SQL Injection is a technique where attackers exploit the SQL “UNION SELECT” statement to combine malicious queries with legitimate ones. Normally, the UNION operator is used to merge results from two different queries. In this attack, the attacker appends a malicious query to the original one to access sensitive information stored in other database tables.

For example, consider the query:
```sql
SELECT Name, Phone, Address FROM Users WHERE Id=1 
UNION ALL SELECT creditCardNumber,1,1 FROM CreditCardTable
```
Here, the attacker forces the database to return not only the user’s details but also credit card numbers from another table. The additional values `1,1` are used just to match the column count of the original query, making it execute successfully.

To check if the database is vulnerable, attackers often add a single quote (`'`) at the end of a URL parameter such as `page.php?id=1'`. If an error message is returned, it reveals that the query structure can be manipulated, signaling a possible UNION SQL injection weakness. Once confirmed, attackers can experiment with UNION clauses to extract sensitive data like usernames, passwords, or financial records.

This makes UNION SQL injection dangerous, as it allows unauthorized access to data by simply chaining queries together, exploiting weak input validation and error handling in web applications.

### Tautology
A tautology-based SQL injection attack works by inserting an OR condition that always evaluates to true, allowing attackers to bypass authentication. For example, in the query:
```sql
SELECT * FROM users WHERE name = '' OR '1'='1';
```
the condition `'1'='1'` is always true, so the database returns all user records, letting the attacker log in without valid credentials.

### End-of-Line Comment 
End-of-line comment SQL injection works by using `--` to ignore part of a query. For example, in
```sql
SELECT * FROM members WHERE username = 'admin'--' AND password = 'password'
```
the `--` makes the database ignore everything after `username = 'admin'`. This allows the attacker to bypass the password check and log in as the admin without knowing the password.

### In-line Comments
In-line comments are a technique used by attackers to make SQL injection more effective by manipulating queries in a way that bypasses security filters. SQL allows the use of comments (such as `/* ... */`) within queries, and attackers exploit this to hide or alter parts of the query while injecting their own malicious input. This makes it harder for simple defenses like input blacklisting or space removal to detect the attack. It can also help them gather information about the database, such as version details, by obfuscating injected code.

For example, consider the query:
```sql
INSERT INTO Users (UserName, isAdmin, Password) 
VALUES ('".$username."', 0, '".$password."')
```
This query is supposed to add a normal user with no admin privileges. However, if an attacker enters:
```sql
UserName = Attacker', 1, /* 
Password = */'mypwd
```
the resulting query becomes:
```sql
INSERT INTO Users (UserName, isAdmin, Password) 
VALUES ('Attacker', 1, /*', 0, '*/'mypwd')
```
Here, the attacker uses an in-line comment (`/* ... */`) to manipulate how the SQL engine interprets the query. The comment essentially cancels out part of the original query, allowing the attacker to set `isAdmin = 1`. This gives them administrator rights instead of being a normal user.

Through this method, attackers can escalate privileges, bypass input validation rules, and gain unauthorized access. Such attacks highlight why input sanitization, parameterized queries, and strict database handling are necessary to defend against SQL injection.

### Piggybacked Query
A piggybacked SQL injection attack happens when an attacker appends a malicious query to a legitimate one, allowing the database to process multiple queries in sequence. This is possible in applications that use batched SQL queries, where the DBMS accepts more than one query at a time. The attacker does not alter the original query but attaches an additional query using a delimiter, usually a semicolon (;). Once the legitimate query executes, the DBMS interprets the delimiter and continues by executing the attacker’s query.

For example, consider the original query:
```sql
SELECT * FROM EMP WHERE EMP.EID = 1001 AND EMP.ENAME = 'Bob'
```
An attacker can inject:
```sql
SELECT * FROM EMP WHERE EMP.EID = 1001 AND EMP.ENAME = 'Bob'; DROP TABLE DEPT;
```
Here, the first part of the query runs as intended, fetching employee details. However, due to piggybacking, the DBMS also executes the malicious `DROP TABLE DEPT;` command, deleting the entire DEPT table.

This technique is dangerous because it can be used to extract sensitive records, alter or delete data, create denial of service by dropping critical tables, or even run remote commands if the DBMS supports them. Since piggybacked queries are executed with the same privileges as the application’s database user, the impact depends on the level of access granted. Weak input validation and failure to use parameterized queries make systems highly vulnerable to this type of attack.

## Error Based SQL Injection 
Error-based SQL injection is a technique where an attacker deliberately triggers database error messages to gather information about the backend system. Instead of directly extracting data, the attacker forces the database to respond with detailed error outputs that reveal useful insights such as database type, table names, or user details. These error messages can then be used to craft more precise and malicious queries.

For example, consider the query:
```sql
SELECT * FROM products WHERE id_product=$id_product
```
If the application executes this query through a request like:
```http
http://www.example.com/product.php?id=10
```
an attacker could inject a payload to force an error, such as:
```http
http://www.example.com/product.php?id=10||UTL_INADDR.GET_HOST_NAME((SELECT user FROM DUAL))--
```
Here, the injected Oracle function `UTL_INADDR.GET_HOST_NAME` attempts to resolve a hostname based on the current database user. Since the database cannot resolve it, it generates an error message such as:
```sql
ORA-292257: host SCOTT unknown
```
This error message reveals the username "SCOTT" from the database. By manipulating the injected query further, attackers can extract other sensitive details such as schema names, table structures, or even passwords.

The strength of error-based SQL injection lies in using the database’s own error-handling responses as a data source. Different database systems (e.g., Oracle, MySQL, SQL Server) expose different levels of detail in their error messages, and attackers exploit these differences to refine their attacks. Ultimately, this method highlights why applications should suppress detailed error outputs and use proper input validation to prevent attackers from turning error messages into a tool for extracting sensitive information.

## UNION SQL Injection 
UNION SQL injection is a technique where an attacker uses the **UNION clause** to merge the results of a malicious query with the results of a legitimate query executed by the application. This allows the attacker to extract sensitive data from other tables in the database that would normally not be exposed.

The process begins by identifying the number of columns in the target query. For example, an attacker may use:
```pgsql
ORDER BY 10--
```
If the query runs without errors, it means the table has at least 10 columns. If it fails with an error like *“Unknown column '10' in 'order clause’”*, the attacker knows the table has fewer than 10 columns. By adjusting the number step by step, the attacker can find the exact number of columns.

After this, the attacker needs to determine the **data types** of each column, because the injected query must match the structure of the original one. For example:
```sql
UNION SELECT 1,null,null--
```
If this executes successfully, it shows the first column accepts integers. By repeating this test with different values, the attacker can map all column types.

Once the structure is known, the attacker can insert a UNION query that pulls sensitive information. Consider the legitimate query:
```pgsql
SELECT Name, Phone, Address FROM Users WHERE Id=$id
```
If the attacker sets `$id` to:
```sql
1 UNION ALL SELECT creditCardNumber,1,1 FROM CreditCardTable
```
The executed query becomes:
```pgsql
SELECT Name, Phone, Address FROM Users WHERE Id=1 
UNION ALL 
SELECT creditCardNumber,1,1 FROM CreditCardTable
```
This merges the original user details with credit card numbers from another table. The result is that sensitive data, like `creditCardNumber`, is exposed through the application’s normal response.

Through this method, attackers can bypass normal query restrictions and extract confidential data such as usernames, passwords, or financial information. This makes UNION SQL injection a serious threat if input validation and secure coding practices are not followed.
