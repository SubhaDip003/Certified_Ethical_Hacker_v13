# 🧹 Clearing Logs
After hiding malicious files using techniques like steganography or NTFS streams to maintain access, attackers focus on **clearing logs** to erase evidence of their activity. This includes deleting event logs, command history, and other forensic traces from the system. By doing this, they cover their tracks and make it difficult for system administrators or investigators to detect the intrusion, helping them stay hidden for longer periods.

---
## Covering Tracks 
Covering tracks is the final and crucial step for attackers to avoid detection after compromising a system. It involves erasing evidence like security logs, failed login records, and unauthorized access traces. Attackers manipulate or delete log files such as SECEVENT.EVT, SYSEVENT.EVT, and APPEVENT.EVT to hide their activities. They may also disable future logging or use tools like rootkits to wipe or alter logs. To remain unnoticed, they restore file attributes like size and timestamps to their original state. While full log deletion often needs admin rights, attackers usually remove only specific entries to avoid suspicion. Detecting such tampering is tough but possible using cryptographic file hashes.

The attacker can manipulate the log files with the help of
- **SECEVENT.EVT** (security): failed logins, accessing files without privileges
- **SYSEVENT.EVT** (system): driver failure, things not operating correctly
- **APPEVENT.EVT** (applications)

### Techniques Used for Covering Tracks

* **Disabling Auditing**: Turns off system monitoring to avoid being tracked.
* **Clearing Logs**: Deletes activity logs to remove evidence.
* **Manipulating Logs**: Alters log entries to hide malicious actions.
* **Network Evasion**: Uses reverse shells, DNS/ICMP tunneling, or TCP tricks to hide traffic.
* **Hiding in OS**: Stores malicious files in **NTFS streams** to avoid detection.
* **Deleting Data Securely**: Uses tools like `cipher.exe` to permanently erase data.
* **Disabling Windows Features**: Shuts off functions like system restore, timestamps, hibernation to remove footprints.
* **Hiding Artifacts**: Blends malicious files within normal OS files or processes.

## Disabling Auditing: Auditpol
🔗Source: [https://learn.microsoft.com]

`Auditpol.exe` is a built-in Windows command-line tool used to manage and control security auditing policies on a system. It allows you to enable, disable, or configure auditing for specific system activities, such as logins, file access, or changes to security settings.

Attackers use this tool during a cyber attack, especially after gaining administrative access, to quietly turn off auditing and avoid detection. By disabling security logs, they can hide their malicious actions, such as stealing data or installing keyloggers. Once their tasks are complete, they may re-enable auditing to reduce suspicion and make it harder for security teams to notice any gaps or unusual activity.

In short, auditpol.exe helps attackers stay hidden by controlling what the system records during their activity.

After gaining access and establishing shell access with the target system, attackers use the following commands to enable/disable system auditing logs: 
```cmd
:: Step 1: View current audit policy settings
C:\>auditpol /get /category:*

:: Step 2: Disable auditing logs (hide attacker activity)
C:\>auditpol /set /category:"system","account logon" /success:disable /failure:disable

:: Step 3: (Optional) Re-enable auditing after actions are done
C:\>auditpol /set /category:"system","account logon" /success:enable /failure:enable
```

## Clearing Logs
**Clear_Event_Viewer_Logs.bat** is a utility that can be used to wipe out the logs of the target system. This utility can be run through command prompt, PowerShell, and using a BAT file to delete security, system, and application logs. Attackers might use this utility to wipe out the logs as one method of covering their tracks on the target system. 

### Steps to clear logs using Clear_Event_Viewer_Logs.bat utility are as follows. 
1. Download the **Clear_Event_Viewer_Logs.bat** utility from [https://www.tenforums.com]
2. Unblock the **.bat** file.
3. Right-click or press and hold on the **.bat** file and click/tap on Run as administrator.
4. If prompted by UAC, click/tap on Yes.
5. A command prompt will now open to clear the event logs. The command prompt will automatically close when finished.

### Steps to clear logs using Meterpreter shell are as follows.
If the system is exploited with Metasploit, the attacker uses a Meterpreter shell to wipe out all the logs from a Windows system: 
1. Launch the **meterpretershell** prompt from the Metasploit Framework.
2. Type `clearev` command in the Meterpreter shell prompt and press Enter. The logs of the target system will start being wiped out.

### Steps to clear PowerShell logs using Clear-EventLog command are as follows.
🔗Source: [https://docs.microsoft.com]

```powershell
# Step 1: Run PowerShell as Administrator
Start-Process powershell -Verb runAs

# Step 2: Clear PowerShell event logs on the local system
Clear-EventLog "Windows PowerShell"
# ➤ Deletes all logs from the PowerShell log (Event ID 4103, 4104, etc.)

# Step 3: Clear multiple log types from local and remote systems
Clear-EventLog -LogName ODiag, OSession -ComputerName localhost, Server02
# ➤ Clears Microsoft Office Diagnostics and Session logs on both local and remote machine

# Step 4: Clear system and application logs with confirmation
Clear-EventLog -LogName Application, System -Confirm
# ➤ Prompts before clearing core system logs to avoid accidental deletion

-ComputerName   # ➤ Clears logs on remote machine(s)
-LogName 	# ➤ Specifies which logs to delete
-Confirm 	# ➤Asks before clearing
-WhatIf 	# ➤Only shows what would happen (used for testing, not shown above)
```

### Steps to clear event logs using wevtutil utility are as follows.
```cmd
:: Step 1: Open Command Prompt as Administrator

:: Step 2: List all available event logs
wevtutil el

:: Step 3: Clear a specific event log (e.g., System log)
wevtutil cl System

:: (Optional) Clear other logs like Application or Security
wevtutil cl Application
wevtutil cl Security
```

## Covering BASH Shell Tracks 
Bash, or the Bourne Again Shell, is a common command-line interface in Linux systems. It keeps a record of every command entered by a user in a hidden file called `.bash_history`. This file can be easily viewed and analyzed to see what actions were performed on the system. While this is useful for system admins, it's a major concern for hackers because investigators can use this file to trace their steps — including what commands they used and when. For this reason, attackers often try to clear or tamper with the bash history to avoid being detected after compromising a system.

```bash
# Step 1: Disable Bash history by setting saved command count to 0
export HISTSIZE=0
# 📌 Prevents the shell from storing any future commands.

# Step 2: Clear the current shell's command history
history -c
# 📌 Erases the history of this session from memory.

# Step 3: Delete the history of only this shell session from disk
history -w
# 📌 Ensures cleared memory history isn't written to the .bash_history file.

# Step 4: Completely delete history from all shells and exit
cat /dev/null > ~/.bash_history && history -c && exit
# 📌 Clears the entire history file and removes command traces from all sessions.

# Step 5: Securely shred (destroy) the history file beyond recovery
shred ~/.bash_history
# 📌 Overwrites the file with random data, making recovery impossible.

# Step 6: Shred + Clear everything and exit
shred ~/.bash_history && cat /dev/null > ~/.bash_history && history -c && exit
# 📌 Final cleanup: shreds, clears, and exits — leaving no trace behind.
```

## Covering Tracks on a Network
### Using Reverse HTTP Shells 
In a reverse HTTP shell attack, the attacker secretly installs malicious code on a victim’s machine that behaves like a web client. This client regularly sends HTTP requests to the attacker’s system, which acts like a web server. Commands are delivered through these requests, executed by the victim, and results are sent back in follow-up requests. Since this traffic looks like regular web browsing, it easily bypasses firewalls and goes unnoticed in the network, helping the attacker stay hidden while controlling the system.

### Using Reverse ICMP Tunnels
Reverse ICMP tunneling is a stealthy technique where attackers hide TCP data inside ICMP echo and reply packets (like ping). Since many firewalls ignore outgoing ICMP traffic, this helps attackers bypass security rules. The attacker sets up a local client, tricks the victim’s machine into sending hidden data in ICMP packets, which reach a proxy server. The proxy extracts the real TCP data and sends it to the attacker, allowing hidden communication or remote access without detection.

### Using DNS Tunneling
DNS tunneling is a technique where attackers hide malicious data or commands inside DNS queries and responses. After compromising an internal system, they use it to connect with an external attacker-controlled server. This creates a secret communication path—called a backchannel—using the DNS protocol. Through this covert path, attackers can send commands or steal sensitive data without being easily detected, as DNS traffic is usually trusted and rarely blocked by firewalls.

###  Using TCP Parameters
TCP parameters can be used by the attacker to distribute the payload and to create covert channels. Some of the TCP fields where data can be hidden are as follows:

* **IP Identification Field**: Used to send one character per packet during an active session. It's simple and bitwise, embedding payloads into each packet.
* **TCP Acknowledgement Number**: Requires a bounce server that relays one hidden character per packet from the victim to the attacker. It’s harder to detect due to indirect routing.
* **TCP Initial Sequence Number (ISN)**: Hides one character per SYN or reset packet without needing a full connection. Great for stealthy, connectionless communication.

These techniques are stealthy and often bypass traditional detection methods.

## Covering Tracks on an OS
### 🪟Windows
NTFS has a feature called ADS that allows attackers to hide a file behind other normal files. Steps to hide files using NTFS are as follows: 
```powershell
# Step 1 – Hide a secret file using NTFS Alternate Data Streams (ADS)
type C:\SecretFile.txt > C:\LegitFile.txt:SecretFile.txt
# ➤ This hides SecretFile.txt inside LegitFile.txt without showing it in Explorer

# Step 2 – Try to access the hidden file (must know its name)
more < C:\LegitFile.txt:SecretFile.txt
# ➤ Shows the hidden content if the stream name is known
```
**Modifying Time**
```powershell
# Modify the file’s timestamp to avoid detection (Timestomp method)
timestomp C:\LegitFile.txt -z "01/01/2020 12:00:00"
# ➤ Changes file's timestamp to an old date (requires timestomp tool)

# Alternate PowerShell method to change LastWriteTime (without external tools)
powershell -Command "(Get-Item 'C:\LegitFile.txt').LastWriteTime = $(Get-Date).AddHours(-10)"
# ➤ Shifts the last modified time 10 hours back using PowerShell
```
### 🐧UNIX/LINUX
In UNIX/Linux, attackers can easily hide files by naming them with a dot (.) at the beginning, like `.hiddenfile`, which makes them invisible to normal `ls` listings. They often drop such files in locations like `/dev`, `/tmp`, or `/etc`. To confuse users further, they might use names like `. ` (dot with a space), which looks like system directories (e.g., `.` or `..`).

They may also edit system log files to erase traces. However, commands they run still get saved in `.bash_history`. To avoid this, smart attackers run `export HISTSIZE=0`, which disables history logging temporarily — helping them stay hidden.

**Modifying Time**
```bash
# Step 1: Change the Access Time (last read time) of a file
touch -a -d '2024-12-31 23:59:59' secret.txt
# This sets the file's "access time" to Dec 31, 2024, 11:59:59 PM

# Step 2: Change the Modification Time (last edited time) of the same file
touch -m -d '2023-01-01 00:00:00' secret.txt
# This sets the file's "modification time" to Jan 1, 2023, 12:00:00 AM
```

## Delete Files using Cipher.exe
`Cipher.exe` is a built-in Windows tool mainly used to encrypt/decrypt files on NTFS drives, but attackers also use it to **securely delete files** so they **can’t be recovered later**. When a malicious file is encrypted, a backup copy is made temporarily. Even if deleted, that backup can be recovered with forensic tools. To hide evidence, attackers use `cipher /w` to overwrite deleted data in three passes:

* **First Pass (0x00)**: Overwrites space with all zeroes.
* **Second Pass (0xFF)**: Overwrites space with all 255s.
* **Third Pass (Random Data)**: Overwrites with random numbers to make recovery nearly impossible.

This method helps attackers erase traces and cover their tracks during an investigation.

The attacker can delete files using Cipher.exe by implementing the following steps:
```cmd
:: Step 1: Open Command Prompt as Administrator
:: Right-click on "cmd" and select "Run as administrator"

:: Step 2: Overwrite deleted files in a specific folder
cipher /w::<drive letter>:\<folder name>

:: Step 3: Overwrite all deleted files on an entire drive
cipher /w:<drive letter>
```

## Disable Windows Functionality 
### Disable the Last Access Timestamp
Attackers may want to hide their tracks by stopping the system from updating file access timestamps. Windows tracks when files are opened or read, but this behavior can be disabled using the `fsutil` tool. By changing the **DisableLastAccess** setting on an NTFS volume, attackers prevent these timestamps from being updated — making it harder to detect if a file was accessed.

* `DisableLastAccess = 1` → Disables last access updates (good for stealth).
* `DisableLastAccess = 0` → Enables last access updates (default tracking behavior).

This trick helps attackers maintain stealth during file access or manipulation on compromised systems.

As shown in the screenshot, attackers use the following command to disable the last access updates:
```powershell
> fsutil behavior set disablelastaccess 1
```

### Disable Windows Hibernation
Disabling Windows hibernation is a stealth tactic used by attackers to **erase the hibernation file (hiberfil.sys)**, which contains a snapshot of system RAM. Since this file can hold sensitive data like credentials, processes, and activities, security teams often analyze it during forensics. By disabling hibernation, attackers ensure that such evidence is wiped out, helping them cover their tracks and avoid detection after compromise.

The attacker can disable Windows hibernation through the registry by implementing the following steps:
- Open **Registry Editor** and navigate to the following location: `Computer\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Power`
- Double-click on `HibernateEnabledDefault` from the right pane; an Edit `DWORD (32-bit) Value` dialog box appears
- In the Value data: field, enter a value of `0` to disable hibernation
- Press OK

**Using PowerShell**
```powershell
# Step 1: Navigate to the Power settings key in Registry
Set-Location 'HKLM:\SYSTEM\CurrentControlSet\Control\Power'

# Step 2: Set 'HibernateEnabledDefault' value to 0 (disabled)
Set-ItemProperty -Path . -Name 'HibernateEnabledDefault' -Value 0
or
powercfg.exe /hibernate off

# Step 3: (Optional) Confirm the value has been set correctly
Get-ItemProperty -Path . | Select-Object HibernateEnabledDefault
```

### Disable Windows Virtual Memory (Paging File)
Disabling Windows Virtual Memory (Paging File) is an important step attackers may take to avoid leaving sensitive data traces. Paging file acts like backup RAM—when real memory is full, Windows stores temporary data (like decrypted files or plain-text passwords) in it.

For example, even after an attacker logs out, decrypted files or sensitive data may still remain in the paging file. Tools or programs may also unintentionally store sensitive info like passwords in it. To avoid forensic recovery, attackers disable it to keep nothing stored on disk after RAM use.

The attacker can disable paging by implementing the following steps: 
1. Open **Control Panel** and navigate to the following location: **System and Security** → **System** → **Advanced system settings**
2. A System Properties dialog box appears; in the **Advanced** tab, click on **Settings**... under the Performance section
3. A Performance Options dialog box appears; go to the **Advanced** tab and click on **Change**... under the Virtual Memory section
4. A Virtual Memory dialog box appears; uncheck **Automatically manage paging file size for all drives**
5. Select the drive where paging should be disabled, then check the option **No paging file* and click **Set**
6. In the **System Properties** window, click **Yes**
7. Finally, click **OK** to implement the changes
