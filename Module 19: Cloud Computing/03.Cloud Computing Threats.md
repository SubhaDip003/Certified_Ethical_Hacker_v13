# 🚨 Cloud Computing Threats
Most organizations adopt cloud technology because it reduces the cost via optimized and efficient computing. Robust cloud technology offers different types of services to end-users; however, many people are concerned about critical cloud security risks and threats, which attackers may take advantage of to compromise data security, gain illegal access to networks, etc. This section deals with significant security threats and vulnerabilities affecting cloud systems

---
## OWASP Top 10 Cloud Security Risks 
🔗Source: [https://owasp.org] <br>
The table below summarizes the top 10 cloud security risks, according to OWASP. 
| **Risks** | **Description** |
|-----------|-----------------|
| **R1 - Accountability and Data Ownership** | - Organizations use the public cloud for hosting business services instead of a traditional data center.<br>- Sometimes using the cloud causes the loss of data accountability and control, whereas using a traditional data center helps in controlling and protecting the data logically and physically.<br>- Using the public cloud can jeopardize data recoverability and result in critical risks, which the organization needs to mitigate promptly. |
| **R2 - User Identity Federation** | - Enterprises use services and applications of different cloud providers, creating multiple user identities and complicating the management of multiple user IDs and credentials.<br>- Cloud providers have less control over the user lifecycle / offboarding. |
| **R3 - Regulatory Compliance** | - Following regulatory compliance can be complex.<br>- Data that is secured in one country may not be secured in another country owing to the lack of transparency and different regulatory laws followed across various countries. |
| **R4 - Business Continuity and Resiliency** | - Performing business continuity in an IT organization ensures that the business can be conducted in a disaster situation.<br>- When organizations use cloud services, there is a chance of risk or monetary loss if the cloud provider handles the business continuity improperly. |
| **R5 - User Privacy and Secondary Usage of Data** | - The use of social websites poses a risk to personal data because they are stored in the cloud and most social application providers mine user data for secondary usage.<br>- The default share feature in social networking sites can jeopardize the privacy of user personal data. |
| **R6 - Service and Data Integration** | - Organizations must ensure proper protection when proprietary data are transferred from the end-user to the cloud data center.<br>- Unsecured data in transit are susceptible to eavesdropping and interception attacks. |
| **R7 - Multi Tenancy and Physical Security** | - Cloud technology uses the concept of multi-tenancy for sharing resources and services among multiple clients, such as networking and databases.<br>- Inadequate logical segregation may lead to tenants interfering with each other’s security features. |
| **R8 - Incidence Analysis and Forensic Support** | - When a security incident occurs, investigating applications and services hosted at a cloud provider can be challenging because event logs are distributed across multiple hosts and data centers located at several countries and governed by different laws and policies.<br>- Owing to the distributed storage of logs across the cloud, law enforcing agencies may face problems in forensics recovery. |
| **R9 - Infrastructure Security** | - Configuration baselines of the infrastructure should comply with the industry best practices because there is constant risk of malicious actions.<br>- Misconfiguration of infrastructure may allow network scanning for vulnerable applications and services to retrieve information, such as active unused ports and default passwords and configurations. |
| **R10 - Non-Production Environment Exposure** | - Non-production environments are used for application design and development and to test activities internally within an organization.<br>- Using non-production environments increases the risk of unauthorized access, information disclosure, and information modification. |

## Top 10 Kubernetes Security Risks (with Mitigation Strategies)

| Risk | Description & Mitigation Strategies |
| :--- | :--- |
| **K01: Insecure Workload Configurations** | - **Description:** Involve deploying applications with settings that increase their vulnerability to attacks. Examples include running containers with **root privileges**, not setting resource limits, or allowing excessive network access. These can be exploited to escalate privileges or cause Denial-of-Service (DoS) attacks. <br><br>- **Mitigation:** Implement best practices like setting appropriate **security contexts** to restrict container privileges, defining **resource limits** to prevent exhaustion, and using network policies to control pod communication. |
| **K02: Supply Chain Vulnerabilities** | - **Description:** Arise from integrating **third-party software and components** into Kubernetes environments (e.g., in container images, application dependencies, or CI/CD pipelines). Attackers can exploit these weaknesses to gain unauthorized access or disrupt services. <br><br> - **Mitigation:** Implement rigorous security measures such as **regularly scanning container images** for vulnerabilities, using **signed and verified images**, and incorporating security checks into the **CI/CD pipeline**. |
| **K03: Overly Permissive RBAC Configurations** | - **Description:** Configurations that grant **excessive permissions** to users and services, which increases the attack surface. Attackers or malicious insiders can exploit these broad permissions to gain unauthorized access or escalate privileges. <br><br> - **Mitigation:** Implement the **principle of least privilege** by carefully defining roles and permissions to restrict access to only what is necessary. |
| **K04: Lack of Centralized Policy Enforcement** | - **Description:** The absence of centralized policy enforcement leads to **inconsistent security policies** across clusters, creating misconfigurations or gaps that make the environment susceptible to attacks. <br><br> - **Mitigation:** Organizations should adopt tools such as **Open Policy Agent (OPA)** to enforce consistent security policies across all clusters. |
| **K05: Inadequate Logging and Monitoring** | - **Description:** The lack of comprehensive and detailed logs makes it difficult to investigate, detect, and respond to security incidents effectively. Security teams become blind to potential threats and malicious activities. <br><br> - **Mitigation:** Implement comprehensive logging and monitoring practices, including capturing detailed logs of all relevant activities, integrating with **centralized logging solutions**, and setting up **real-time alerting** for suspicious activities. |
| **K06: Broken Authentication Mechanisms** | - **Description:** Weaknesses in the process of verifying the identity of users and services. This can include improper configuration of authentication protocols, weak password policies, or failure to implement **Multi-Factor Authentication (MFA)**. <br><br>- **Mitigation:** Implement strong authentication practices that include configuring robust authentication protocols, enforcing strong password policies, and **enabling MFA for all users**. |
| **K07: Missing Network Segmentation Controls** | - **Description:** The lack of proper isolation between different components and workloads within the cluster, which makes it easier for attackers to **move laterally** within the environment once they gain initial access. <br><br> - **Mitigation:** Implement **network policies** to control traffic between pods and services and use third-party solutions to enforce segmentation to help limit lateral movement. |
| **K08: Secrets Management Failures** | - **Description:** Occur when sensitive information (passwords, API keys, certificates) is **improperly stored or handled**, such as hardcoding secrets in application code or storing them in unsecured locations. <br><br>- **Mitigation:** Use **dedicated secrets management tools**, ensuring secrets are encrypted, and strictly controlling access to them. |
| **K09: Misconfigured Cluster Components** | - **Description:** Incorrect settings in core Kubernetes components (API server, etcd, kubelet) often stemming from default settings or lack of security hardening, leaving the cluster vulnerable. Attackers can exploit these to gain control. <br><br> - **Mitigation:** **Regularly audit** cluster configurations, apply security best practices, and keep all core components up-to-date. |
| **K10: Outdated and Vulnerable Kubernetes Components** | - **Description:** Using outdated Kubernetes and its dependencies, which may contain **known vulnerabilities** that can be exploited by attackers to compromise the cluster. <br><br>- **Mitigation:** Establish a crucial routine for **regularly updating** Kubernetes and its associated components to maintain a secure environment. |

## OWASP Top 10 Serverless Security Risks 
🔗Source: [https://owasp.org] <br>
OWASP Top 10 Serverless Security Risks highlight that while serverless computing removes server management, it does not eliminate application-level threats. Vulnerabilities in the code can still lead to attacks such as SQL injection, XSS, DoS, and broken authentication. This means serverless apps face many of the same risks as traditional web applications, requiring secure coding and strong security practices.

## OWASP Top 10 Serverless Security Risks

| Risks | Attack Vector | Security Weakness | Impact |
| :--- | :--- | :--- | :--- |
| **A1 – Injection** | Input arrives not only from API but also from serverless functions that are invoked from various event sources, such as cloud storage events (S3 Blob), stream data processing (AWS Kinesis), database modifications (DynamoDB, CosmoDB), code modifications (AWS CodeCommit), and notifications (SMS, email, IoT). Firewall cannot filter the events generated through email or a database. | SQL/NoSQL injection <br> OS command injection <br> Code injection | Impact depends on the permissions of the vulnerable function. If the function has access to the cloud storage, the injected code can **delete data** or **upload corrupted data**. |
| **A2 – Broken Authentication** | Serverless functions are stateless, executed separately, have different resources, and are triggered by different events. Attackers try to identify missing resources, such as open APIs and public cloud storage. If functions are invoked through organizational emails, attackers can send spoofed emails to trigger the functions and execute internal functionality without authentication. | Poor design of identity and access controls | **Accessing functions without authentication** leads to sensitive data leakage, system business logic breakage, and execution flow disruption. |
| **A3 – Sensitive Data Exposure** | Attacks on traditional web applications, such as cracking keys, man-in-the-middle (MITM) attacks, and data stealth in transit and at rest, are also applicable to serverless applications. Attackers target cloud storage (S3, Blob) and database tables (DynamoDB, CosmoDB). | Storing sensitive data in **plaintext** or using **weak encryption**. <br> Writing data to the `/tmp` directory without removing after use. | **Exposure of sensitive data**, such as PII, health records, credentials, and credit card details. |
| **A4 – XML External Entities (XXE)** | If serverless functions are running inside internal virtual private networks (VPNs), attacks such as scanning internal networks and DoS are not possible. These attacks only affect the designated container in which the function is running. | Using XML processors might make the application vulnerable to XXE attacks. | **Leakage of function code and sensitive files** (environment variable, `/tmp` directory, etc.). |
| **A5 – Broken Access Control** | The stateless nature of serverless architecture allows attackers to exploit **over-privileged functions** to gain unauthorized access to resources. | Granting functions **access and privileges to unnecessary resources**. | Impact depends on the compromised resource. <br> **Leakage of data** from cloud storage and database. |
| **A6 – Security Misconfiguration** | Misconfigured functions with a **long timeout** and **low concurrency limit** allow attackers to perform **DoS attacks**. | Functions with **long timeout configuration** and **low concurrency**. <br> Poor patch management. | **Sensitive information leakage**, loss of money, DoS, and unauthorized access to cloud resources. |
| **A7 – Cross-Site Scripting (XSS)** | In traditional applications, XSS vulnerabilities arrive from databases or reflective inputs, but in serverless applications, they also arrive from sources such as emails, logs, cloud storage, IoT, etc. | **Untrusted input** used to generate data without proper **escaping**. | **User impersonation**. <br> Access to sensitive data, such as API keys. |
| **A8 – Insecure Deserialization** | Dynamic languages (e.g., Python, NodeJS) along with JavaScript object notation (JSON), a serialized datatype, allow attackers to perform deserialization attacks. | **Deserialization vulnerabilities** in Python, JavaScript, etc. | Impact depends on the sensitivity of data the application handles. <br> **Running arbitrary code**, data leakage, and resource and account control. |
| **A9 – Using Components with Known Vulnerabilities** | Serverless functions are used for microservices, which depend on **third-party libraries** for execution. Vulnerable third-party libraries allow attackers to gain an entry point to serverless applications. | Lack of knowledge on component-heavy deployment patterns. | Business impact depends on the specification of **known vulnerabilities**. |
| **A10 – Insufficient Logging and Monitoring** | Complex serverless auditing and **lack of monitoring** and **timely response** pave the way for various attacks. | Insufficient security monitoring and auditing. | The impact of **late security incident identification** can be significant. |
