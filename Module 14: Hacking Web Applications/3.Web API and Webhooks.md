# 🔗 Web API and Webhooks
Web APIs allow applications to exchange data with multiple online sources and are widely used in platforms like social media, e-commerce, and search engines. Webhooks enable applications to send real-time data to other systems. Because APIs often handle sensitive personal or business data, any security weakness can lead to serious breaches, making strong API security measures essential.

---
## Web API 
A Web API is an interface that allows applications to communicate over the internet using protocols like HTTP. It provides tools and functions to retrieve or update data from multiple sources without complexity. By centralizing data and business logic, a Web API simplifies updates, improves consistency, and reduces the need to modify each client application separately.

## Web Service APIs 
1. **SOAP API** → A protocol that lets different systems (Windows, macOS, Linux, etc.) talk to each other using XML over HTTP, often used for creating, updating, or deleting data like profiles or credentials.
2. **REST API** → An architectural style for web services that uses simple, stateless communication between systems, making it easy to request and change web resources.
3. **RESTful API** → A web service following REST rules, using HTTP methods (GET, POST, PUT, DELETE) to improve speed, scalability, and reliability of applications. <br>
   APIs with the following features can be referred to as to RESTful APIs:

    - **Stateless** → The server doesn’t remember your previous requests; the client stores session data.
    - **Cacheable** → The client can store API responses to speed up future requests.
    - **Client-Server Environment** → The client (front end) and server (back end) work separately but communicate through requests.
    - **Uniform Interface** → Every resource has a unique URL, and you can use standard methods like GET, POST, PUT, DELETE to interact with it.
    - **Layered System** → Data can pass through multiple layers (like cache servers) without the client knowing exactly where it came from.
    - **Code on Demand** → The server can send temporary code to the client to add or change features.

5. **XML-RPC** → A simpler protocol than SOAP that sends data in XML format, using less bandwidth.
6. **JSON-RPC** → Similar to XML-RPC but sends data in lightweight JSON format for faster communication.

## Webhooks 
Webhooks are user-defined HTTP callbacks that send data to other applications in real time when specific events occur, such as posting a comment or updating code. They act like “reverse APIs,” automatically pushing information instead of waiting for requests. Common uses include sending alerts, messages, or notifications—like informing users when an out-of-stock item becomes available.

## Operation of Webhooks
Webhooks are mechanisms that send automated notifications to a specified URL when certain events occur. Once registered through a domain’s interface or API, they execute predefined code whenever the event is triggered. The receiving system can identify the event, its source, time, and details via an HTTP POST request, allowing it to react accordingly. Webhooks can be set as private or public.

### Webhooks vs. APIs 
- Webhooks are automated messages from websites to the server. APIs are used for server-to-website communication.
- Webhooks get reports or notifications via HTTP POST only when a new update is made. APIs make calls irrespective of the data updates.
- Webhooks update applications or services with real-time information. API needs additional implementations to perform this activity.
- Webhooks have less control over data flow. APIs have easy control over data flow.

## OWASP Top 10 API Security Risks 
Source: [https://owasp.org]

| API | Risk | Descriptions |
|---|---|---|
| API1 | **Broken Object-Level Authorization** | - APIs can have security issues when they expose endpoints that use object IDs without proper checks, making it easier for attackers to target them. <br> - Hackers can change the object ID in a request to access data they shouldn’t be allowed to see. <br> - This can lead to sensitive data being exposed, lost, or altered by unauthorized users. |
| API2 | **Broken Authentication** | - When authentication is not set up correctly, attackers can steal login tokens or take advantage of flaws in the system to pretend to be other users. <br> - Once they do this, they can take full control of the victim’s account, read private information, and perform sensitive actions as if they were the real user. |
| API3 | **Broken Object Property Level Authorization** | - Developers might send all data fields of an object to the user without checking which ones are sensitive, relying on the user to hide them. <br> - This can allow attackers to view, change, or delete private or important data. <br> - In some cases, it can let attackers gain higher access rights or even fully take over an account. |
| API4 | **Unrestricted Resource Consumption** | - Attackers can initiate multiple concurrent requests using automated tools designed to cause DoS via high loads of traffic, impacting APIs’ service rate. <br> - Attackers find APIs that do not limit client interactions or resource consumption and craft API requests including parameters that control the number of resources. |
| API5 | **Broken Function-Level Authorization** | - Complexity in access control policies through different hierarchies, groups, and roles between administrative and regular functions can cause authorization errors. <br> - Allow attackers to gain unauthorized access to administrative functions or users’ resources. |
| API6 | **Unrestricted Access to Sensitive Business Flows** | - Some APIs let users perform important actions like buying tickets or posting comments without limiting how often they can be done, which attackers can abuse. <br> - Hackers figure out which parts of the API handle these actions and how they connect to complete the process. <br> - They study how the business works through the API, find valuable functions, and then automate them to cause financial or operational damage. |
| API7 | **Server-Side Request Forgery** | - A server-side request forgery (SSRF) flaw enables an attacker to coerce the application to send a crafted request to an unexpected destination, even when protected by a firewall or a VPN. <br> - Attackers exploit this vulnerability by finding an API endpoint that accesses a URI provided by the client. <br> - Exploitation might lead to internal services enumeration (e.g., port scanning), information disclosure, bypassing firewalls, or other security mechanisms |
| API8 | **Security Misconfiguration** | - Attackers will often attempt to find unpatched flaws, common endpoints, services running with insecure default configurations, or unprotected files and directories to gain unauthorized access or knowledge of the system. <br> - Attackers use automated tools to detect and exploit misconfigurations such as unnecessary services or legacy options. <br> - Security misconfigurations not only expose sensitive user data but also system details that can lead to a full server compromise. |
| API9 | **Improper Inventory Management** | - Proper inventory management of hosts and deployed API versions is important to mitigate issues. <br> - Attackers usually gain unauthorized access through old API versions or endpoints left running unpatched and using weaker security requirements. <br> - Attackers can gain access to sensitive data or even take over the server when different API versions/deployments are connected to the same database with real data. |
| API10 | **Unsafe Consumption of APIs** | - Developers tend to trust data received from third-party APIs more than user input and thus tend to adopt weaker security standards. <br> - Attackers go after integrated third-party services instead of trying to compromise the target API directly. <br> - Successful exploitation may lead to exposure of sensitive information to unauthorized actors, different types of injections, or denial of service. |
