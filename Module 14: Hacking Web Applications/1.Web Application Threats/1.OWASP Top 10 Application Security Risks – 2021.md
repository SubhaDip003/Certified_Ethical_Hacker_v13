OWASP is an international organization that maintains a list of the top 10 vulnerabilities and flaws of web applications. The latest OWASP top 10 application security risks are as follows. 
1. **A01 – Broken Access Control** →
   This happens when users can access things they shouldn’t, like someone else’s account or admin features. Example: changing a URL to view another user’s data. <br>
   Attacks that fall under this category include:
   - Directory Traversal
   - Hidden Field Manipulation

3. **A02 – Cryptographic Failures** →
   This is when sensitive data (like passwords or credit card info) isn’t properly encrypted. If it's sent in plain text or uses weak encryption, attackers can steal it. <br>
   Attacks that fall under this category include:
   - Cookie Snooping
   - RC4 NOMORE Attack
   - Same-Site Attack
   - Pass-the-Cookie Attack

5. **A03 – Injection** →
   When an application accepts untrusted input and runs it as code. Attackers can use this to run malicious commands. Example: SQL Injection lets them steal or change data. <br>
    Attacks that fall under this category include:
   - SQL Injection
   - Command Injection
   - LDAP Injection
   - Cross-Site Scripting (XXS)
   - Buffer Overflow

6. **A04 – Insecure Design** →
   Flaws in how an app is planned or designed, like missing security checks. It leads to attacks like CAPTCHA bypass or session hijacking. <br>
   Attacks that fall under this category include:
   - Business Logic Bypass Attack
   - Web-based Timing Attacks
   - CAPTCHA Attacks
   - Platform Exploits

8. **A05 – Security Misconfiguration** →
   This happens when systems are not properly set up. Using default passwords, showing error messages with too much info, or not updating software can make apps easy to hack. <br>
   Attacks that fall under this category include:
   - XML External Entity (XXE) Attack
   - Unvalidated Redirects and Forwards
   - Directory Traversal
   - Hidden Field Manipulation

10. **A06 – Vulnerable and Outdated Components** →
   Using old software parts (like plugins, libraries, etc.) with known bugs lets attackers exploit them. These weak parts often run with full access and can lead to data theft. <br>
   Attacks that fall under this category include:
      - Platform Exploits
      - Magecart Attack
      - Buffer Overflow

12. **A07 – Identification and Authentication Failures** →
   Weak login systems can be bypassed. Attackers may guess passwords or hijack sessions if tokens and credentials aren't protected properly. <br>
   Attacks that fall under this category include:
      - Cross-Site Request Forgery
      - Cookie/Session Poisoning
      - Cookie Snooping

14. **A08 – Software and Data Integrity Failures** →
   If software updates or data are not verified properly, attackers can inject malware. For example, insecure auto-update features or untrusted plugins. <br>
   Attacks that fall under this category include:
      - Insecure Deserialization
      - Unvalidated Redirects and Forwards
      - Watering Hole Attack
      - Denial-of-Service (DoS)
      - Buffer Overflow
      - Web Service Attacks
      - Platform Exploits
      - Magecart Attack

16. **A09 – Security Logging and Monitoring Failures** →
   If the app doesn’t log activities or alert about unusual actions (like failed logins), attacks can go unnoticed. This gives hackers more time to do damage. <br>
   Attacks that fall under this category include:
      - Web Service Attacks

18. **A10 – Server-Side Request Forgery (SSRF)** →
    When an app fetches a URL without checking if it's safe, attackers can use it to access internal systems behind firewalls or steal sensitive info. <br>
    Attacks that fall under this category include:
       - Injecting an SSRF Payload
       - Cross-Site Port Attack (XSPA)
       - DNS Rebinding Attack
       - H2C Smuggling Attack

## A01 – Broken Access Control
Broken Access Control happens when a web app doesn’t properly restrict what users can do. Attackers can exploit this flaw to bypass restrictions and gain access to functions or data meant only for admins or other users. This lets them view, change, or delete information they shouldn’t be able to. It usually occurs due to poor testing and missing security checks in the code.

According to OWASP 2021 R3 revision, the common vulnerabilities associated with access control are as follows: 
1. **Too Much Access by Default** → Users get more permissions than needed, instead of getting only what they require (least privilege not enforced).

2. **Bypassing Access Controls** → Attackers change URLs, parameters, or use tools to reach restricted pages or functions they shouldn't access.

3. **Account Takeover via ID** → Hackers guess or manipulate user IDs to access or change someone else’s account details.

4. **Unprotected API Methods** → Sensitive API actions like PUT, POST, or DELETE are exposed without proper access control checks.

5. **Privilege Escalation** → A normal user can become an admin or gain higher access by exploiting flaws in the system.

6. **Tampering with Metadata** → Attackers change cookies, hidden fields, or JWTs to bypass access restrictions or raise their privileges.

7. **CORS Misconfiguration** → APIs are accessed from unauthorized websites due to poor Cross-Origin Resource Sharing settings.

8. **Force Browsing** → Attackers directly visit admin or user-only URLs without logging in or after logging in as a low-privilege user.

## A02 – Cryptographic Failures/Sensitive Data Exposure 
Cryptographic failures, also known as sensitive data exposure, happen when web applications do not properly protect important data like passwords, credit card numbers, or personal information. This can occur due to weak encryption, insecure storage, or poor handling of cryptographic keys. Even if data is encrypted, attackers can still steal it if outdated or flawed algorithms like MD5 or SHA-1 are used, or if keys and certificates are stored in unsafe locations. Reusing weak initialization vectors or using bad encryption modes can also make data easier to break. To prevent these attacks, developers must use strong, up-to-date encryption, securely store keys, and ensure error messages don’t leak useful information.

## A03 – Injection Flaws
Injection flaws happen when a web application takes untrusted input and runs it as part of a command or query. Attackers use this to send harmful commands—like in SQL, LDAP, or XPath—that the application mistakenly executes. This can lead to data theft, data loss, or even full control over the system. These flaws are common in old or poorly written code and can be found using scanners or fuzzers. When exploited, attackers can read, change, or delete sensitive data by tricking the application into running their malicious input.

There are many types of injection flaws, some of which are discussed below: 
1. **SQL Injection** → Attackers inject harmful SQL code into input fields (like login forms) to access or steal data from the website's database without permission.

2. **Command Injection** → Hackers find weak spots in input handling and insert system commands that get executed on the server, which can lead to full system compromise.

3. **LDAP Injection** → Attackers insert malicious input into LDAP queries to gain unauthorized access or modify directory information (like login permissions or user data).

4. **Cross-Site Scripting (XSS)** → Hackers inject malicious scripts into web pages that run in users’ browsers, allowing them to steal sessions, deface websites, or redirect users to harmful sites.

## A04 – Insecure Design 
Insecure design happens when an application is built without proper security measures in place. This often results from developers not fully understanding security risks or skipping risk analysis during planning. Such flaws can lead to serious issues like SQL injection or exposed cloud storage. Attackers take advantage of this by analyzing how the app works to find and exploit these weaknesses in its design or structure.

The following are some exploitations that can be performed by software and data integrity failures:
- Request forgery
- Authentication hijacking
- Identity theft
- Data loss
- DoS attacks

**Example** <br>
Attackers often attempt to exploit poorly implemented APIs, which fail to filter requests properly. They search for weak APIs that are not integrated with security gateways to differentiate malicious inputs. Then, they attach malicious code to a vulnerable API. When a user accesses that API via an application, the malicious code loads along with the database content on the user’s browser.
