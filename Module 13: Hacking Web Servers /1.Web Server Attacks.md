# 💥 Web Server Attacks
An attacker can use many techniques to compromise a web server, such as DoS/DDoS, Domain Name System (DNS) server hijacking, DNS amplification, directory traversal, man in the middle (MITM)/sniffing, phishing, website defacement, web server misconfiguration, HTTP response splitting, web cache poisoning, Secure Shell (SSH) brute force, and web server password cracking. This section describes these attack techniques in detail. 

---
## DNS Server Hijacking 
The Domain Name System (DNS) resolves a domain name to its corresponding IP address. A user queries the DNS server with a domain name, and the DNS server responds with the corresponding IP address. <br>
**DNS Server Hijacking** is when an attacker takes control of a DNS server and changes its settings. So, when a user types a real website address, they're secretly redirected to a fake, attacker-controlled site. This trick is used to steal information or spread malware.

## DNS Amplification Attack 
Recursive DNS query is a method of requesting DNS mapping. The query goes through DNS servers recursively until it fails to find the specified domain name to IP address mapping. <br> 
The following are the steps involved in processing recursive DNS requests; these steps are illustrated in the below figure.
- **Step 1:** Users who desire to resolve a domain name to its corresponding IP address send a DNS query to the primary DNS server specified in its Transmission Control Protocol (TCP)/IP properties.
- **Steps 2 to 7:** If the requested DNS mapping does not exist on the user’s primary DNS server, the server forwards the request to the root server. The root server forwards the request to the .com namespace, where the user can find DNS mappings. This process repeats recursively until the DNS mapping is resolved.
- **Step 8:** Ultimately, when the system finds the primary DNS server for the requested DNS mapping, it generates a cache for the IP address in the user’s primary DNS server.

<img width="721" height="323" alt="image" src="https://github.com/user-attachments/assets/ed50f069-80a5-4464-ba8b-dffe0e78a46f" />

Attackers exploit recursive DNS queries to perform a DNS amplification attack that results in DDoS attacks on the victim’s DNS server. <br>
The following are the steps involved in a DNS amplification attack; these steps are illustrated in the below figure.
- **Step 1:** The attacker instructs compromised hosts (bots) to make DNS queries in the network.
- **Step 2:** All the compromised hosts spoof the victim’s IP address and send DNS query requests to the primary DNS server configured in the victim’s TCP/IP settings.
- **Steps 3 to 8:** If the requested DNS mapping does not exist on the victim’s primary DNS server, the server forwards the requests to the root server. The root server forwards the request to the .com or respective top-level domain (TLD) namespaces. This process repeats recursively until the victim’s primary DNS server resolves the DNS mapping request.
- **Step 9:** After the primary DNS server finds the DNS mapping for the victim’s request, it sends a DNS mapping response to the victim’s IP address. This response goes to the victim because bots use the victim’s IP address. The replies to copious DNS mapping requests from the bots result in DDoS on the victim’s DNS server.

## Directory Traversal Attacks 
**Directory Traversal Attack** is a web attack where hackers access restricted folders on a server by using tricks like `../` in the URL. This happens when a web app or server doesn’t properly validate user input. Attackers can then reach sensitive files outside the web root, potentially stealing or executing them. Poor server configuration or unpatched code often causes this vulnerability.

## Website Defacement
**Website defacement** is when a hacker breaks into a web server and changes how a website looks. They may add unwanted images, text, or popups, or even replace the whole website. This is done by injecting malicious code to alter the site’s content without permission.

<img width="519" height="392" alt="image" src="https://github.com/user-attachments/assets/85499140-9dbd-42ad-9ca0-0e58caa6c92b" />

They often use methods like SQL injection to break in. Defaced sites may show false messages or propaganda, and can also infect visitors’ computers with viruses. It damages the organization’s image and puts users at risk.

## Web Server Misconfiguration 
Web server misconfiguration refers to the configuration weaknesses in web infrastructure that can be exploited to launch various attacks on web servers, such as directory traversal, server intrusion, and data theft. The following are some web server misconfigurations:
- Verbose debug/error messages
- Anonymous or default users/passwords
- Sample configuration and script files
- Remote administration functions
- Unnecessary services enabled
- Misconfigured/default SSL certificates

### Examples of Web Server Misconfigurations 
“**Keeping the server configuration secure requires vigilance**”—Open Web Application Security Project (OWASP) <br> 
**Web Server Misconfigurations** happen when admins set up a server incorrectly, leaving security gaps. These flaws—like weak settings, exposed files, or vulnerable scripts—can let attackers access sensitive data, bypass login systems, or even take full control of a website. Regular checks and proper configurations are key to keeping servers secure.

## HTTP Response-Splitting Attack 
**HTTP Response Splitting** is a web attack where the attacker injects **new line characters** into the **response headers** of a vulnerable web server, like Apache. This makes the server split one HTTP response into **two separate responses**.

The attacker crafts a request with malicious header data. If the server **doesn’t validate input properly**, it includes the attacker’s input in the response header. This tricks the browser into processing **two HTTP responses**—the first one controlled by the attacker.

This can lead to:

* **Redirecting users** to fake websites
* **Launching other attacks** like XSS, CSRF, or SQL injection

The core issue is **poor input validation**, which allows the attacker to break the response format and insert harmful content.

### Example of an HTTP Response-Splitting Attack 
In this example, the attacker sends a response-splitting request to the web server. The server splits the response into two and sends the first response to the attacker and the second response to the victim. After receiving the response from the web server, the victim requests service by providing credentials. Simultaneously, the attacker requests for the index page.
Subsequently, the web server sends the response to the victim’s request to the attacker, and the victim remains uninformed.
